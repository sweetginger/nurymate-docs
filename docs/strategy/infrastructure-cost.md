## 1. 설계 원칙 및 가정

### 1.1 왜 소비자 수 기준인가

기존 인프라 설계는 셀러(판매자) 수를 기준으로 했으나, 실제 서버 부하는 **소비자(구매자)의 행동**에 의해 결정된다.

| 부하 원인 | 발생 주체 |
|----------|---------|
| 상품 페이지 조회 | 소비자 |
| 장바구니·결제 요청 | 소비자 |
| 이미지·정적 자산 요청 | 소비자 |
| 상품 검색·필터 | 소비자 |
| 재고 확인 API | 소비자 |
| 관리자 대시보드 조회 | 셀러 (소량) |
| 상품 등록·수정 | 셀러 (소량) |

셀러는 관리자 행동이 주이며, 소비자 대비 트래픽 비중이 극히 낮다. **인프라 sizing의 핵심 변수는 소비자 수**다.

### 1.2 가정값

| 항목 | 초기 (Phase 1) | 성장기 (Phase 2+) | 근거 |
|------|-------------|----------------|------|
| 셀러 1명당 소비자 수 | 500명 | 2,000명 | 국내 소형 D2C 쇼핑몰 평균 구독자·팔로워 기준 보수 추정 |
| 소비자 월 방문 횟수 | 3회 | 3회 | 이커머스 평균 재방문율 참고 |
| 주문 전환율 | 3% | 3% | 이커머스 업계 평균 2~4% 중간값 |
| 세션당 평균 API 호출 | 15회 | 15회 | 상품 목록 + 상세 + 검색 + 결제 플로우 합산 추정 |
| 평균 상품 이미지 크기 | 200 KB | 200 KB | WebP 압축 후 기준 |
| DB 읽기:쓰기 비율 | 90:10 | 90:10 | 이커머스 표준 |

### 1.3 트래픽 산정 공식

```
월 세션 수 = 소비자 수 × 월 방문 횟수
월 API 호출 수 = 월 세션 수 × 세션당 평균 API 호출
월 주문 수 = 월 세션 수 × 전환율
```

## 3. 비용 요약 및 단가 분석

### 3.1 시나리오별 비용 요약표

| 시나리오 | 셀러 수 | 소비자 수 | 월 인프라 비용 (USD) | 월 인프라 비용 (KRW) |

| B | 100명 | 50,000명 | $580 | ₩800,200 |
| C | 500명 | 250,000명 | $2,425 | ₩3,346,500 |

### 3.2 성장에 따른 비용 효율성

| 구간 | 셀러 수 증가배 | 비용 증가배 | 비용 효율 |
|------|------------|----------|---------|
| A → B | 10배 | 5.6배 | **규모의 경제 발생** (캐싱·CDN 효과) |
| B → C | 5배 | 4.2배 | **효율 지속 개선** |

셀러(소비자) 수가 10배 늘어도 인프라 비용은 5~6배 수준으로 억제된다. CloudFront CDN과 ElastiCache 캐싱이 원점 DB 요청을 흡수하기 때문이다.

## 4. BEP 계산기 연계 (셀러당 인프라 비용)

### 4.1 셀러당 월 인프라 비용

| 시나리오 | 셀러 수 | 월 인프라 비용 (KRW) | 셀러당 월 인프라 비용 |
|---------|--------|-------------------|-------------------|
| A | 10명 | ₩143,500 | **₩14,350** |
| B | 100명 | ₩800,200 | **₩8,002** |
| C | 500명 | ₩3,346,500 | **₩6,693** |

### 4.2 수익 구조에서의 의미

셀메이트CM 수익 모델(슬라이딩 수수료, pricing-model.md 참조)과 연계하면:

| 시나리오 | 셀러당 인프라 비용 | 셀러당 월 최소 수수료 수익 목표 | 커버리지 |
|---------|-----------------|--------------------------|---------|
| A (초기) | ₩14,350/월 | ₩50,000/월 (셀러 월 매출 5백만 원 × 1% 수수료 기준) | 인프라 커버 가능 |
| B (성장) | ₩8,002/월 | ₩50,000/월 | 여유 마진 확보 |
| C (스케일) | ₩6,693/월 | ₩50,000/월 | 고수익 구간 진입 |

> 셀러당 인프라 비용이 셀러 수 증가에 따라 **₩14,350 → ₩6,693으로 53% 하락**. 스케일업이 수익성 개선의 핵심 레버다.

### 4.3 손익 분기점 기준

초기(시나리오 A) 셀러당 인프라 비용 ₩14,350을 커버하려면:
- **슬라이딩 수수료 1% 기준**: 셀러 월 매출 **₩1,435,000 이상** 필요
- **슬라이딩 수수료 2% 기준**: 셀러 월 매출 **₩717,500 이상** 필요
- 국내 소형 D2C 쇼핑몰 월 매출 평균이 수백만 원대임을 감안하면, 시나리오 A에서도 인프라 비용 커버는 현실적이다.

## 5. 비용 절감 전략

### 5.1 즉시 적용 가능 (초기 단계)

| 전략 | 예상 절감 | 비고 |
|------|---------|------|
| **ECS Fargate → EC2 Reserved Instance 전환** | 30~40% | 1년 약정 시. 트래픽 예측 가능 시점부터 전환 |
| **RDS Reserved Instance 1년 약정** | 30~40% | 온디맨드 대비 절감. 시나리오 B 이상 권장 |
| **S3 Intelligent-Tiering** | 10~20% | 자주 접근하지 않는 이미지 자동 저비용 계층 이동 |
| **CloudFront 캐시 히트율 최적화** | 20~30% | TTL 튜닝으로 원점 요청 감소. 오리진 비용 절감 |

### 5.2 중기 전략 (시나리오 B 이상)

| 전략 | 예상 절감 | 비고 |
|------|---------|------|
| **읽기 전용 API → Read Replica 라우팅** | DB 비용 20% | 현재 아키텍처(backend/db/orm.py)에 이미 Read/Write 분리 구현됨 |
| **ElastiCache 캐시 히트율 90%+ 유지** | API 비용 30% | 상품 목록·재고 조회는 캐시 우선 처리 |
| **이미지 CDN 최적화 (WebP + 사이즈별 캐싱)** | S3+CDN 25% | 프론트엔드 이미 WebP 압축 구현(s3Upload.ts) |
| **Savings Plans (Compute)** | 전체 컴퓨팅 20% | ECS + Lambda 혼용 시 효과적 |

### 5.3 장기 아키텍처 최적화 (시나리오 C 이상)

| 전략 | 내용 |
|------|------|
| **멀티테넌시 DB 최적화** | 셀러별 스키마 분리 vs 공유 테이블 트레이드오프 재검토 |
| **엣지 함수 (CloudFront Functions)** | 간단한 리다이렉트·인증 처리를 엣지로 이동하여 ECS 부하 경감 |
| **비동기 작업 오프로드 (SQS + Lambda)** | 이메일 발송·주문 후처리 등 비동기 처리로 ECS 동기 부하 감소 |
| **데이터 웨어하우스 분리** | 분석 쿼리를 운영 DB에서 분리 (Redshift 또는 Athena) |
