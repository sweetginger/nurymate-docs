# 누리메이트 API 공개 및 써드파티 연동 허용 정책 [4-22]

> 작성일: 2026-03-10 | 버전: v0.1
> 관련 태스크: [4-9] 서드파티 앱 마켓플레이스 미수용 방침과 정합성 검토

---

## 1. 정책 배경

### 1.1 핵심 질문

"누리메이트가 외부 개발자에게 API를 공개하고 써드파티 툴 연동을 허용해야 하는가?"

이 질문은 [4-9]에서 결론 내린 "서드파티 앱 마켓플레이스 미수용 방침"과 얼핏 충돌해 보이지만, 사실 다른 차원의 결정이다.

| 구분 | 방침 |
|------|------|
| **앱 마켓플레이스 (외부 앱 판매)** | [4-9] 결론: 초기엔 미수용. 써드파티가 기능을 파는 마켓 구조는 자체 개발 집중 후 검토 |
| **API 공개 (연동 허용)** | 이 문서의 논의 대상. 판매자가 자체 시스템과 연동하거나 에이전시가 헤드리스 구현 가능 |

→ **API 공개 ≠ 앱 마켓플레이스**. API를 공개하면서 앱 마켓을 운영하지 않는 것이 가능하다.

## 2. 경쟁사 API 정책 벤치마크

### 카페24 오픈 API

- **공개 범위**: 상품·주문·고객·배송·통계 API 전면 공개
- **접근 방식**: OAuth 2.0 인증, 판매자 동의 기반
- **활용 사례**: ERP 연동, 재고 관리 시스템, 마케팅 자동화 도구 등
- **제한**: API 호출 한도 (일일 10,000회 기본), 실시간 이벤트 웹훅 제공
- **파트너 프로그램**: 앱 개발자 등록 → 앱 스토어 입점 → 매출 쉐어
- **누리메이트 시사점**: 카페24 판매자가 사용 중인 ERP·OMS와의 연동 호환성이 마이그레이션 결정에 영향을 줌. 동일 API 커버리지 확보가 전환 장벽 제거에 필수

### Shopify API (Admin API, Storefront API)

- **공개 범위**: Admin API (전체 스토어 관리), Storefront API (구매자 경험), Partner API (파트너 관리)
- **접근 방식**: OAuth 앱 또는 커스텀 앱 방식
- **특징**: GraphQL 기반, 실시간 웹훅, 헤드리스 커머스 완전 지원
- **제한**: API 버전 관리 (분기별 업데이트, 1년 지원 사이클)
- **누리메이트 시사점**: 헤드리스 강점을 살리려면 Storefront API급 완성도의 공개 API 필수. 에이전시가 커스텀 프론트엔드를 구현할 수 있어야 함

### WooCommerce REST API

- **공개 범위**: 완전 오픈소스, REST API 전면 공개
- **특징**: 무인증 또는 Basic Auth / OAuth, 커뮤니티 기반 확장
- **누리메이트 시사점**: 오픈소스 수준의 완전 공개는 지식재산 리스크. 제한적 공개 구조가 적합

## 3. API 공개 이점 및 리스크

### 이점

| 이점 | 설명 |
|------|------|
| **생태계 확장** | 에이전시·개발자가 누리메이트 기반 서비스 구축 → 고객 확보 채널 다각화 |
| **고객 락인** | 판매자가 자체 ERP·OMS와 연동할수록 플랫폼 전환 비용 증가 |
| **개발 속도 가속** | 자체 개발팀이 모든 통합을 담당하지 않아도 됨 |
| **마이그레이션 용이성** | 카페24에서 이전 시 기존 연동 ERP를 재활용 가능 |
| **헤드리스 구현** | 디자인 에이전시가 커스텀 프론트엔드 구현 → 고부가가치 고객 유치 |

### 리스크

| 리스크 | 설명 | 대응 방안 |
|--------|------|---------|
| **API 남용** | 과도한 호출로 서버 부하 | Rate limit + API 키별 호출 쿼터 |
| **데이터 보안** | 판매자 데이터 무단 접근 | OAuth 2.0 + 세분화된 권한 스코프 |
| **서비스 품질 저하** | 써드파티 버그가 판매자 스토어에 영향 | 써드파티 앱은 샌드박스 격리, 누리메이트 책임 범위 명확화 |
| **API 버전 의존성** | 오래된 API 버전 유지 부담 | 명확한 지원 사이클 공지 (최소 18개월 지원) |

## 4. 단계별 API 개방 전략

### Phase 0 — 내부 API (런칭 전)

- 외부 공개 없음
- 헤드리스 프론트엔드 구현을 위한 내부 API 설계 완성
- API 설계 원칙 확립: RESTful + 향후 GraphQL 전환 고려

### Phase 1 — 파트너 한정 공개 (런칭 ~ 2026 Q4)

**대상**: 계약된 에이전시 파트너, 파트너 프로그램 등록 개발자

**공개 범위**:
- Storefront API: 상품 조회, 카테고리, 장바구니, 결제 초기화
- 판매자 한정 Admin API: 주문 관리, 상품 CRUD, 재고 조회
- 웹훅: 주문 완료, 결제 완료, 재고 변경

**접근 방법**: 파트너 신청 → 심사 → API 키 발급

**목적**: 헤드리스 에이전시 파트너 확보, 마이그레이션 도구 개발 지원

### Phase 2 — 판매자 셀프서비스 API (2027 Q1)

**대상**: 자체 연동 필요 판매자 (ERP, OMS 연동 등)

**공개 범위 확장**:
- 고객 데이터 API (동의 기반)
- 통계/분석 API (읽기 전용)
- 마케팅 API (이메일·쿠폰 트리거)

**수익화 옵션**: 고급 API 접근권 월 구독 (5~10만 원/월, 프리미엄 기능 구독 패키지 포함)

### Phase 3 — 오픈 생태계 (2027 Q3+)

**조건**: 플랫폼 GMV 월 50억 원 이상, 개발자 커뮤니티 형성 확인 후

**검토 항목**:
- 공개 API 문서 포털 운영
- 써드파티 앱 허용 정책 재검토 ([4-9] 재검토 조건)
- 개발자 마켓플레이스 또는 파트너 디렉토리

## 5. [4-9] 서드파티 앱 미수용 방침과 정합성

| 구분 | 현재 방침 | API 공개 후 변화 |
|------|---------|---------------|
| 써드파티 앱 마켓플레이스 | Phase 1~2 미수용 | Phase 3에서 재검토 |
| 파트너 에이전시 헤드리스 구현 | Phase 1부터 허용 | API 공개로 가능 |
| ERP·OMS 연동 (판매자 자체 시스템) | Phase 2부터 허용 | Admin API로 가능 |
| 써드파티 앱이 누리메이트 내 실행 | 미수용 유지 | — |

**결론**: API 공개는 [4-9] 방침과 충돌하지 않는다. Phase 1~2에서는 "판매자 자체 시스템 연동"과 "에이전시 헤드리스 구현"에 한정하고, "앱 마켓플레이스 외부 앱 판매" 모델은 Phase 3 이후 별도 검토한다.

---

## 6. API 설계 원칙

1. **판매자 동의 기반 데이터 접근**: 모든 API는 판매자가 명시적으로 승인한 범위만 접근 가능 (OAuth 스코프)
2. **최소 권한 원칙**: 각 API 키는 필요한 최소한의 스코프만 부여
3. **Rate Limiting**: 기본 10,000회/일, 프리미엄 100,000회/일
4. **버전 관리**: `/v1/`, `/v2/` 명시적 버전 분리, 최소 18개월 지원
5. **보안 감사**: 분기별 API 접근 로그 검토, 이상 접근 자동 차단
