# AI 채팅 인터페이스 기반 설정 기능 기획서

> 작성일: 2026-03-06
> 관련 태스크: [6-1] AI 채팅으로 어드민 설정 변경, [6-2] 기능 전수 목록화
> 버전: v0.1 (초안)

## 1. 개요 및 목표

### 1.1 배경

카페24의 최대 약점 중 하나는 복잡한 어드민 UI다. 배송 정책 하나를 바꾸려면 5단계 메뉴를 거쳐야 하고, 처음 쓰는 판매자는 어디서 무엇을 바꿔야 할지 조차 찾기 어렵다. 누리메이트는 이 문제를 AI 채팅 인터페이스로 해결한다.

자연어로 명령하면 AI가 인텐트를 파악하고, 확인 후 설정을 변경한다. 판매자는 메뉴 구조를 외울 필요가 없다.

### 1.2 벤치마크 사례

| 서비스 | 방식 | 특징 |
|--------|------|------|
| **Shopify Sidekick** | 자연어 → 설정 변경 실행 | Shopify Flow 자동화 생성, 다국어 지원 20개 언어, 권한 기반 접근 제어 |
| **Notion AI** | 콘텐츠 Q&A + 커넥터 | 워크스페이스 내 자연어 검색·편집, 출처 인용 |
| **Linear AI** | 이슈 자동 생성·분류 | 자연어 → Jira 티켓, 우선순위 자동 설정 |
| **Intercom Fin** | AI 1차 대응 + 에스컬레이션 | 외부 시스템(CRM, 재고) 연동 후 액션 실행 |
| **Microsoft 365 Copilot** | GUI+CUI 융합 | 플로팅 채팅 레이어가 기존 UI 위에 오버레이 |

**핵심 교훈**: Shopify Sidekick이 가장 직접적인 경쟁/레퍼런스. 자연어 → Shopify Flow 워크플로 생성이 킬러 기능. 누리메이트는 "확인 → 실행" 패턴과 롤백을 더 강화해 신뢰도를 높인다.

### 1.3 목표

- **판매자 목표**: 메뉴 탐색 없이 채팅으로 모든 설정 변경 가능
- **제품 목표**: AI 설정 채팅을 통한 어드민 DAU 30% 증가, 설정 완료율 50% 향상
- **차별화 목표**: "카페24보다 10배 쉬운 어드민" 포지셔닝

## 2. 핵심 사용 시나리오

### 2.1 시나리오 예시 (대화 흐름)

#### 시나리오 A: 배송비 정책 변경

```
판매자: 5만원 이상 무료배송으로 바꿔줘

AI: 현재 설정을 확인했어요.
    - 현재: 무료배송 조건 없음 (항상 3,000원)
    - 변경 후: 5만원 이상 구매 시 무료배송, 미만 시 3,000원

    이대로 변경할까요? [변경하기] [취소]

판매자: 변경하기

AI: 완료했어요. 배송비 정책이 업데이트됐습니다.
    - 적용 범위: 국내 배송 전체
    - 적용 시점: 즉시 (현재 진행 중인 주문 제외)
    변경 내역 보기 | 되돌리기 (24시간 이내)
```

#### 시나리오 B: 상품 일괄 가격 수정

```
판매자: 티셔츠 카테고리 상품 전부 10% 할인 적용해줘

AI: 티셔츠 카테고리 상품 23개를 찾았어요.
    - 현재 가격 범위: 29,000원 ~ 89,000원
    - 변경 후 가격 범위: 26,100원 ~ 80,100원 (10% 할인가)

    할인 방식을 선택해주세요:
    [정가 10% 할인] [할인가 별도 설정] [세일 배너 자동 추가]

판매자: 정가 10% 할인, 세일 배너도 붙여줘

AI: 알겠어요. 다음 작업을 함께 처리할게요:
    1. 티셔츠 23개 상품가 10% 인하
    2. 각 상품 상단에 "SALE 10% OFF" 배너 적용

    진행할까요? [확인] [취소]
```

#### 시나리오 C: 마케팅 자동화 설정

```
판매자: 장바구니 담고 구매 안 한 사람한테 1시간 후에 카카오 알림 보내줘

AI: 장바구니 이탈 자동화를 설정할게요.
    - 조건: 장바구니 담기 후 1시간 내 미결제
    - 액션: 카카오 알림톡 발송
    - 메시지 초안: "안녕하세요! 장바구니에 담아두신 상품이 아직 기다리고 있어요."

    카카오 알림톡이 연동되어 있지 않아요.
    연동 먼저 진행할까요? [카카오 연동하기] [나중에]
```

#### 시나리오 D: 모호한 요청 처리

```
판매자: 스토어 좀 예쁘게 해줘

AI: 조금 더 구체적으로 알려주시면 도와드릴 수 있어요.
    다음 중 어떤 것이 필요하신가요?

    [테마 색상 변경] [폰트 바꾸기]
    [메인 배너 수정] [상품 진열 방식 변경]

    또는 원하시는 것을 자유롭게 말씀해 주세요.
```

### 2.2 설정 가능 범위 (Phase별)

| Phase | 기간 | 범위 |
|-------|------|------|
| Phase 1 (MVP) | 출시 ~ 3개월 | 스토어 기본 설정, 배송 정책, 결제 ON/OFF |
| Phase 2 | 3~6개월 | 상품 일괄 관리, 할인/쿠폰, 마케팅 자동화 트리거 |
| Phase 3 | 6~12개월 | 리포트 커스텀, 고객 세그먼트 설정, 워크플로 조합 |

## 4. AI 처리 구조

### 4.1 자연어 → 인텐트 파싱

```
사용자 입력
    │
    ▼
[전처리 레이어]
  - 언어 감지 (한국어/일본어/영어)
  - 맥락 히스토리 포함 (직전 5개 메시지)
    │
    ▼
[인텐트 분류 LLM]
  - Intent: CHANGE_SETTING / QUERY / AMBIGUOUS / OUT_OF_SCOPE
  - Domain: delivery / product / payment / marketing / report / store
  - Action: update / create / delete / toggle / bulk_update
  - Target: 변경 대상 리소스 ID/명칭
  - Value: 변경 값 또는 조건
    │
    ▼
[파라미터 추출기]
  - 누락 파라미터 감지 → 추가 질문 생성
  - 현재 설정 값 조회 (읽기 세션)
  - 변경 영향 범위 계산
```

**인텐트 분류 예시**

| 입력 | Intent | Domain | Action | Value |
|------|--------|--------|--------|-------|
| "무료배송 5만원으로" | CHANGE_SETTING | delivery | update | 50000 |
| "어제 매출 얼마?" | QUERY | report | read | date=yesterday |
| "예쁘게 해줘" | AMBIGUOUS | store | - | - |
| "직원 채용해줘" | OUT_OF_SCOPE | - | - | - |

### 4.2 확인 → 실행 플로우

```
인텐트 파싱 완료
    │
    ├── QUERY → 즉시 조회 결과 반환 (확인 불필요)
    │
    └── CHANGE_SETTING
            │
            ▼
        [영향도 판단]
        - LOW: 공지사항, 메모, 알림 설정
        - MED: 배송비, 상품 노출, 쿠폰 생성
        - HIGH: 가격 일괄 변경, PG사 변경, 결제 수단 삭제
            │
            ├── LOW → 변경 내역 미리보기 표시 → [즉시 적용]
            │
            ├── MED → 변경 전/후 비교 카드 표시 → [변경하기] [취소]
            │
            └── HIGH → 변경 영향 상세 안내 → 텍스트 확인 입력 요구
                       예: "계속하려면 '확인'을 입력해주세요"
                            │
                            ▼
                        [실행 레이어]
                          - 쓰기 세션으로 DB 업데이트
                          - 변경 로그 기록 (rollback_snapshot 포함)
                          - 실시간 스토어 반영 확인
                            │
                            ▼
                        완료 메시지 + 되돌리기 버튼 (24시간)
```

### 4.3 에러 처리 및 롤백

| 에러 유형 | 처리 방식 | 복구 방법 |
|----------|----------|---------|
| 파라미터 누락 | AI가 추가 질문 생성 | 대화 계속 |
| 권한 없음 | "이 설정은 [어드민/소유자]만 변경할 수 있어요" | 권한자 태그 알림 |
| 비즈니스 규칙 위반 | "최소 배송비는 0원 이상이어야 해요" | 유효 범위 안내 |
| API 실패 | "일시적 오류가 발생했어요. 다시 시도할까요?" | 자동 재시도 (3회) |
| 부분 실패 (일괄 처리) | "23개 중 21개 성공, 2개 실패. 실패 목록 보기" | 선택적 재시도 |
| 사용자 롤백 요청 | 변경 전 스냅샷으로 복구 | 24시간 이내 |

**롤백 스냅샷 정책**
- 모든 CHANGE_SETTING 실행 전 현재 설정 값 snapshot 저장
- 저장 기간: 24시간 (HIGH 영향도는 72시간)
- 롤백 방법: 완료 메시지의 "되돌리기" 버튼 또는 "방금 설정 되돌려줘" 명령

## 5. UX 설계 원칙

### 5.1 채팅 인터페이스 배치

- **항상 노출**: 어드민 우측 하단 플로팅 버튼 (기존 UI 위에 오버레이)
- **단축키**: `Ctrl/Cmd + K` 즉시 활성화 (Linear 스타일)
- **컨텍스트 인식**: 현재 보고 있는 페이지 기준으로 기본 도메인 설정
  - 배송 설정 페이지에서 열면 → 배송 관련 명령 우선 처리

### 5.2 응답 UI 패턴

| 응답 유형 | UI 컴포넌트 |
|----------|------------|
| 조회 결과 | 텍스트 + 인라인 테이블/차트 |
| 설정 변경 확인 | 변경 전/후 비교 카드 + CTA 버튼 |
| 선택지 제시 | 빠른 응답 버튼 (최대 4개) |
| 오류 안내 | 붉은 테두리 카드 + 해결 방법 링크 |
| 완료 안내 | 녹색 체크 + 변경 내역 링크 + 되돌리기 |

### 5.3 메시지 설계 원칙

- **친근하지만 명확하게**: 반말 존댓말 혼용 금지, 일관된 "~해요" 체
- **숫자는 구체적으로**: "배송비가 바뀌었어요" (X) → "배송비가 3,000원 → 무료(5만원 이상)로 바뀌었어요" (O)
- **항상 되돌리기 제공**: 모든 변경 완료 메시지에 24시간 롤백 옵션 포함
- **모호함은 즉시 해소**: AMBIGUOUS 인텐트는 4개 이하의 선택지로 유도

## 6. 구현 우선순위 (MVP vs 고도화)

### MVP (출시 필수, Phase 1)

| 기능 | 이유 |
|------|------|
| 자연어 인텐트 분류 (CHANGE/QUERY/AMBIGUOUS) | 핵심 엔진 |
| 확인 → 실행 → 롤백 플로우 | 신뢰성 기반 |
| 스토어 기본 설정 10개 변경 | 데모 가능 범위 |
| 배송비/무료배송 조건 변경 | 가장 빈번한 요청 |
| 일 매출 / 상품별 랭킹 조회 | 즉각적 가치 제공 |
| 플로팅 채팅 UI + 단축키 | 접근성 |

### Phase 2 고도화

| 기능 | 이유 |
|------|------|
| 상품 일괄 가격 변경 | 고위험 → 충분한 확인 UX 필요 |
| 쿠폰/할인 자동화 설정 | 마케팅 자동화 연동 필요 |
| 장바구니 이탈 자동화 | 카카오/SMS 연동 선행 |
| 다국어 지원 (일본어) | 일본 시장 Phase 2 진입 시 |

### Phase 3 고도화

| 기능 | 이유 |
|------|------|
| 워크플로 조합 ("A하면 B하게 해줘") | 복합 인텐트 처리 고도화 |
| 리포트 커스텀 생성 | 분석 레이어 별도 구축 필요 |
| 고객 세그먼트 동적 생성 | CRM 고도화 선행 |

## 8. Shopify식 설정 화면 연동 vs 직접 실행 비교 (관련 태스크: [17-2])

### 8.1 Shopify Sidekick의 전환 사례

#### 초기 방식 (2024): 설정 화면 연동 (Navigation + Context Handoff)

Shopify Sidekick은 초기에 "채팅 → 해당 설정 화면 이동 → 화면 내 채팅 상호작용 → 미리보기 → 반영" 흐름으로 설계되었다. 판매자가 "배송비 바꿔줘"라고 입력하면 AI가 배송 설정 화면으로 이동시키고, 해당 화면에서 채팅 패널이 현재 설정값을 읽어 변경 가이드를 제공하는 구조였다.

**문제점**:
- **단계 수 과다**: 채팅 입력 → 화면 이동 → 화면 재확인 → 설정값 입력 → 저장의 최소 5단계가 기존 UI 직접 조작과 차이가 없었다
- **컨텍스트 단절**: 채팅 창과 설정 화면 간 상태 공유가 불완전해, 화면 이동 후 채팅 맥락이 끊기는 경험이 빈번했다
- **인지 부하 증가**: 채팅 UI와 기존 어드민 UI를 동시에 파악해야 하므로 초보 판매자의 혼란이 가중됐다
- **결과**: 사용률 저조 — 채팅을 써도 결국 직접 UI를 조작하는 패턴으로 회귀

#### 개선 방식 (2025): 직접 실행 + Human-in-the-Loop

2025년 Shopify는 Sidekick을 "채팅 요청 → AI가 직접 실행 (단, 모든 실행 전 판매자 승인 필수)" 방식으로 전환했다. 화면 이동 없이 채팅 안에서 변경 전/후 미리보기를 인라인으로 보여주고, 판매자가 승인하면 즉시 반영된다.

**전환 근거**:
- 판매자가 원하는 것은 "설정 화면 탐색 보조"가 아니라 "설정 변경 완료"임을 사용 데이터로 확인
- Human-in-the-Loop 원칙(모든 실행 전 승인 요구)으로 신뢰 문제를 해결하면서 UX 효율성 확보
- 채팅 내 인라인 미리보기가 별도 화면 이동 없이도 충분한 변경 확인 수단이 됨을 검증

---

### 8.2 두 방식 비교

| 항목 | 화면 연동 방식 | 직접 실행 방식 |
|------|--------------|--------------|
| **UX 효율성** | 낮음 — 화면 이동 포함 5단계 이상 | 높음 — 채팅 안에서 2단계 (확인 → 완료) |
| **신뢰도** | 중간 — 판매자가 직접 확인 후 저장하므로 명시적이나, 흐름이 복잡 | 높음 — 실행 전 변경 전/후 비교 + 승인 단계로 투명성 확보 |
| **구현 복잡도** | 높음 — 딥링크, 화면-채팅 상태 바인딩, 실시간 설정값 동기화 필요 | 중간 — API 직접 호출, 인라인 미리보기 UI, 롤백 스냅샷 필요 |
| **실수 방지** | 중간 — 판매자가 화면에서 최종 확인하지만 UI 조작 실수 가능 | 높음 — AI가 변경값 명시 + 롤백 24시간 보장 |
| **초보 판매자 적합성** | 낮음 — 화면 구조를 어느 정도 알아야 이해 가능 | 높음 — 채팅 결과만 확인하면 됨, 메뉴 구조 불필요 |
| **실패 패턴** | 채팅-화면 간 컨텍스트 단절, 중간 이탈률 높음 | AI 오판 실행 (롤백으로 복구), 승인 피로감 |
| **확장성** | 설정 화면마다 바인딩 구현 필요 → 기능 추가 시 공수 누적 | API 레이어 확장으로 대응 가능 → 기능 추가 공수 일정 |

---

### 8.3 설정 화면 연동 방식 기술 구현 요소

참고 목적으로 화면 연동 방식의 구현 요소를 정리한다 (누리메이트 채택 방식 아님).

#### 딥링크 + 채팅 상태 공유

```
채팅 인텐트 분류 완료 (Domain: delivery, Action: update)
    │
    ▼
딥링크 URL 생성: /admin/settings/delivery?chat_context={session_id}&intent=update_free_shipping
    │
    ▼
해당 설정 화면으로 라우팅 (채팅 패널 유지)
    │
    ▼
설정 화면이 URL 파라미터에서 chat_context 추출
    → 채팅 상태 스토어(예: Zustand/Redux)에서 현재 인텐트 읽기
    → 화면 내 관련 입력 필드 하이라이팅
```

- 채팅 컨텍스트 스토어와 페이지 라우터가 session_id를 공유 키로 상태 동기화
- 화면 이동 후 채팅 패널이 현재 URL 경로를 감지해 도메인 컨텍스트 자동 전환

#### 설정 화면이 채팅 패널에 현재 설정값을 실시간으로 제공하는 방식

```
설정 화면 마운트 시
    → 해당 화면의 설정 엔티티 ID를 채팅 컨텍스트에 등록
    → 채팅 패널이 구독(subscribe) 시작

설정값 변경 이벤트 발생 (사용자 입력 또는 AI 제안 반영)
    → 설정 화면 컴포넌트가 변경값을 채팅 컨텍스트 스토어에 emit
    → 채팅 패널이 실시간으로 변경 전/후 표시 업데이트
```

#### 인라인 미리보기 UI 구조

- 채팅 응답 버블 안에 "변경 전 → 변경 후" 카드 컴포넌트 삽입
- 설정 화면의 해당 필드를 채팅 버블과 시각적으로 연결하는 하이라이트 오버레이
- [적용] 클릭 시 설정 화면이 폼 제출(submit) 이벤트를 트리거

#### 적용/취소 확인 단계

- 채팅 패널의 CTA 버튼([적용] / [취소])이 설정 화면의 저장/리셋 액션과 바인딩
- 취소 시: 설정 화면 필드값을 초기 로드 값으로 복원

### 8.4 누리메이트 권장 방향

**결론**: **직접 실행 방식** 채택 (Human-in-the-Loop 승인 단계 필수 유지)

**근거**:

1. **타겟 사용자 특성**: 누리메이트의 핵심 타겟은 "카페24를 어렵다고 느끼는 판매자"다. 이들에게 화면 연동 방식은 기존 어드민 UI를 이해해야 한다는 점에서 차별화가 약하다. 채팅 안에서 완결되는 경험이 "카페24보다 10배 쉬운 어드민" 포지셔닝에 부합한다.

2. **Shopify의 실패 검증**: 화면 연동 방식은 이미 사용률 저조가 검증된 패턴이다. 동일한 실패를 반복할 이유가 없다.

3. **구현 복잡도 대비 효과**: 화면 연동은 설정 화면마다 딥링크·상태 바인딩 구현이 필요해 초기 공수가 높고 유지보수 부담이 크다. 직접 실행은 API 레이어 중심으로 구현해 기능 확장 시 일관된 패턴 적용이 가능하다.

4. **신뢰 확보 방안 충분**: Human-in-the-Loop(모든 변경 전 승인 요구) + 롤백 스냅샷 24시간 보장으로 "AI가 멋대로 바꿀 수 있다"는 불신을 해소할 수 있다.

**구현 방안**:

- **변경 전/후 비교 카드 인라인 표시** (섹션 5.2 응답 UI 패턴에 이미 반영됨): 화면 이동 없이 채팅 버블 안에서 변경 내용 시각화
- **영향도별 승인 단계 차등 적용** (섹션 4.2 확인 → 실행 플로우): LOW는 미리보기 후 즉시 적용, MED는 [변경하기] 버튼, HIGH는 텍스트 확인 입력
- **화면 연동 방식의 부분 활용 허용 (보조적)**: 판매자가 "배송 설정 화면 보여줘"처럼 명시적으로 요청할 때만 해당 화면으로 이동하는 딥링크 기능은 Phase 2에서 보조 기능으로 구현 가능. 단, 기본 흐름은 직접 실행 유지.
- **승인 피로 방지**: 동일 세션 내 반복 변경(예: 여러 상품 가격 순차 변경)은 "일괄 확인" 모드 제공 — Phase 2 검토 항목
