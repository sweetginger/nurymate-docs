# POS 연동 기획서

> 작성일: 2026-03-31 | 버전: v0.1

---

## 1. 개요

### 1.1 한 줄 요약

셀메이트 스캔페이·POS를 1순위로 연동하고, 이후 도도포스·포스뱅크 등 타사 POS로 확장하여 셀메이트CM 오프라인 재고·주문·결제·고객 데이터를 단일 플랫폼에서 통합 관리할 수 있게 한다.

### 1.2 왜 POS 연동인가

POS 연동은 단순 편의 기능이 아닌 **락인(lock-in) 기제**다. 오프라인 매장의 일별 매출·재고 데이터가 셀메이트CM로 흐르는 순간, 판매자는 플랫폼 교체 비용이 급격히 높아진다. 특히 1차 타겟인 셀메이트 온오프라인 몰 디자이너 브랜드는 셀메이트 POS(스캔페이)를 이미 사용 중이므로, 셀메이트 스캔페이 연동만으로도 즉시 락인 효과를 확보할 수 있다.

### 1.3 연동 대상 정의

| 분류 | 대상 | 우선순위 |
|------|------|---------|
| **1순위** | 셀메이트 스캔페이 (Scanpay) | P0~P1 |
| **2순위** | 도도포스 (DodoPOS) | P2 |
| **3순위** | 포스뱅크 (POSBank) | P2 |
| **4순위** | 얼리페이 (EarlyPay) / 기타 | P3 |

## 2. 기능 목록 및 상세 명세

### 2.1 셀메이트 스캔페이 연동

#### 2.1.1 연동 방식

**선택: API + Webhook 혼용**

| 방향 | 방식 | 사용 목적 |
|------|------|---------|
| 셀메이트CM → 스캔페이 | REST API (Push) | 상품 정보·재고 전달, 가격 업데이트 |
| 스캔페이 → 셀메이트CM | Webhook (Event Push) | POS 판매 발생 시 재고 차감 알림, 결제 완료 이벤트 |
| 셀메이트CM → 스캔페이 | REST API (Pull) | 매장 재고 조회, 일별 판매 집계 조회 |

**API vs Webhook 선택 근거**

- 실시간성이 필요한 이벤트(POS 판매 발생 → 재고 차감)는 Webhook이 지연 최소화
- 셀메이트CM가 먼저 데이터를 밀어야 하는 경우(상품 등록·가격 변경)는 REST API Push
- 재고 조회처럼 요청-응답 구조가 자연스러운 경우는 REST API Pull

#### 2.1.2 공유 데이터 범위

**① 재고 (Inventory)**

| 데이터 | 방향 | 설명 |
|--------|------|------|
| 상품(SKU) 목록 | 셀메이트CM → 스캔페이 | POS에서 판매할 상품 목록 동기화 |
| 재고 수량 | 양방향 | 온라인 주문 차감 ↔ 오프라인 판매 차감 통합 |
| 가격 정보 | 셀메이트CM → 스캔페이 | 온라인 가격 변경 시 POS 자동 반영 |
| 재고 부족 알림 | 셀메이트CM → 판매자 | 특정 SKU 재고 임계값 도달 시 알림 |

**② 주문 (Order)**

| 데이터 | 방향 | 설명 |
|--------|------|------|
| 클릭앤콜렉트 픽업 대기 주문 | 셀메이트CM → 스캔페이 POS | POS 화면에 픽업 대기 목록 노출 |
| 픽업 완료 처리 | 스캔페이 → 셀메이트CM | POS에서 픽업 처리 시 셀메이트CM 주문 상태 갱신 |
| 오프라인 판매 주문 이력 | 스캔페이 → 셀메이트CM | 통합 주문 대시보드에 오프라인 판매 포함 |

**③ 결제 (Payment)**

| 데이터 | 방향 | 설명 |
|--------|------|------|
| 오프라인 결제 금액·수단 | 스캔페이 → 셀메이트CM | 통합 정산 대시보드 (`unified-payment-dashboard.md`) 연계 |
| 환불 처리 이력 | 스캔페이 → 셀메이트CM | 온오프라인 환불 통합 집계 |

**④ 고객 (Customer)**

| 데이터 | 방향 | 설명 |
|--------|------|------|
| 고객 포인트 적립·차감 이벤트 | 양방향 | 오프라인 결제 시 포인트 적립, 포인트 사용 |
| 쿠폰 사용 검증 | 셀메이트CM → 스캔페이 | 온라인 발급 쿠폰을 POS에서 사용 시 유효성 확인 |
| 고객 ID 연결 | 스캔페이 ↔ 셀메이트CM | 전화번호 또는 OAuth 기반 온오프라인 고객 통합 |

### 2.2 타사 POS 연동

#### 2.2.1 도도포스 (DodoPOS)

- **연동 가능성**: 높음. 도도포스는 오픈 API를 제공하며 외부 연동 사례(배달의민족, 네이버 주문 등)가 있음
- **주요 기능 범위**: 재고 연동, 주문 수신 (클릭앤콜렉트 픽업 알림)
- **공식 API 문서**: https://dodo-pos.com/api (파트너사 별도 계약 필요)
- **우선순위 조건**: 셀메이트CM 입점 판매자 중 도도포스 사용자 비율 5% 이상 확인 후 착수

#### 2.2.2 포스뱅크 (POSBank)

- **연동 가능성**: 중간. 포스뱅크는 하드웨어 중심 POS 업체로 API 공개 범위가 제한적
- **연동 방식 대안**: 포스뱅크 측 파트너십 협의 → SDK 또는 전용 커넥터 방식 검토
- **주요 기능 범위**: 재고 동기화에 한정 (주문 연동은 협의 필요)
- **우선순위 조건**: 셀메이트 기존 파트너사 여부 확인 후 결정

#### 2.2.3 얼리페이 (EarlyPay) 및 기타

- 얼리페이는 결제 단말기 중심 서비스로 POS 연동보다 **결제 정산 데이터 연동**이 주목적
- 국내 POS 시장 점유율 상위 업체: 포스피아, 한국NFC, 퍼스트포스 등도 장기적으로 검토
- 연동 방식은 셀메이트 스캔페이 연동 완료 후 표준 POS 어댑터(Adapter) 패턴으로 확장

#### 2.2.4 타사 POS 연동 우선순위 매트릭스

| POS | 시장 점유율 | API 개방성 | 셀메이트 연계성 | 우선순위 |
|-----|-----------|-----------|--------------|---------|
| 스캔페이 (셀메이트) | 타겟 셀러 직접 사용 | 내부 협의 가능 | 최고 | **P1** |
| 도도포스 | 외식·패션 중간 점유율 | 오픈 API 있음 | 중간 | **P2** |
| 포스뱅크 | 제조업·소매 중간 점유율 | 제한적 | 낮음 | **P2** |
| 얼리페이 | 결제 단말 중심 | 결제 데이터 중심 | 낮음 | **P3** |
| 기타 | - | - | - | **P3** |

## 3. 사용자 플로우

### 3.1 판매자 — POS 연동 설정 플로우

```
[판매자 — 셀러 어드민]
    │
    ├─ 1. 설정 > 오프라인 매장 > POS 연동
    │
    ├─ 2. POS 브랜드 선택
    │      [셀메이트 스캔페이] [도도포스] [포스뱅크] [기타]
    │
    ├─ 3. 셀메이트 스캔페이 선택 시
    │      → 스캔페이 계정 로그인 (OAuth 방식)
    │      → 연동할 매장 선택 (스캔페이에 등록된 매장 목록)
    │      → 연동 데이터 범위 선택 (재고/주문/포인트 각각 토글)
    │      → 연동 테스트 → 완료
    │
    ├─ 4. 연동 완료 후
    │      → 매장별 재고 실시간 조회 활성화
    │      → 클릭앤콜렉트 설정 가능 상태로 전환
    │
    └─ 5. 연동 상태 모니터링
           → 마지막 동기화 시각 표시
           → 오류 발생 시 알림 배너 + 재연결 버튼
```

### 3.2 오프라인 판매 → 셀메이트CM 재고 반영 플로우

```
[오프라인 매장 직원]
    │
    ├─ 1. 스캔페이 POS에서 상품 바코드 스캔 → 결제 처리
    │
    ├─ 2. 스캔페이 → 셀메이트CM Webhook 전송
    │      POST /api/v1/webhooks/scanpay
    │      { event: "pos.sale_completed", sku_id, qty, store_id, ... }
    │
    ├─ 3. 셀메이트CM 재고 원장 차감
    │      → inventory_ledger에 OFFLINE / delta: -1 기록
    │
    ├─ 4. 온라인 상품 상세 페이지 재고 수량 실시간 갱신
    │      → 해당 매장 "재고 1개" → "품절" 전환
    │
    └─ 5. 재고 임계값(예: 0개) 도달 시
           → 판매자 알림 (어드민 알림 + 카카오 채널)
```

## 4. 기술 구현 방향

### 4.1 POS 연동 API 엔드포인트 목록

#### 셀메이트CM가 제공하는 엔드포인트 (POS → 셀메이트CM)

```
POST   /api/v1/webhooks/scanpay              POS 이벤트 수신 (판매·재고 변동)
POST   /api/v1/webhooks/pos/{provider}       타사 POS 이벤트 수신 (provider: dodopos / posbank)
GET    /api/v1/pos/products                  POS가 조회하는 상품·재고 목록
GET    /api/v1/pos/pickup-orders             픽업 대기 주문 목록 (매장 ID 필터)
PATCH  /api/v1/pos/pickup-orders/{order_id}  픽업 완료 처리
```

#### 셀메이트CM가 호출하는 스캔페이 API (셀메이트CM → 스캔페이)

```
POST   {SCANPAY_API}/products/sync           상품·재고 정보 푸시
GET    {SCANPAY_API}/stores/{store_id}/inventory  매장 재고 조회
POST   {SCANPAY_API}/coupons/validate        쿠폰 유효성 검증
POST   {SCANPAY_API}/points/event            포인트 이벤트 전송 (적립·차감)
```

#### POS 어댑터 패턴 설계

타사 POS 확장을 고려해 Provider 별 어댑터(Adapter) 레이어를 백엔드에 둔다.

```
backend/domain/pos/
├── adapter/
│   ├── base.py          POSAdapter 추상 클래스
│   ├── scanpay.py       셀메이트 스캔페이 구현체
│   ├── dodopos.py       도도포스 구현체 (P2)
│   └── posbank.py       포스뱅크 구현체 (P2)
├── service.py           POS 연동 비즈니스 로직
├── repository.py        POS 연동 설정 저장
└── model.py             POSConnection, POSSaleEvent SQLModel
```

### 4.2 데이터 동기화 주기 및 방식

| 데이터 | 동기화 방식 | 주기 | 비고 |
|--------|-----------|------|------|
| 재고 차감 (오프라인 판매) | Webhook (실시간) | 즉시 | POS 판매 이벤트 발생 시 |
| 픽업 완료 처리 | Webhook (실시간) | 즉시 | POS 픽업 처리 이벤트 |
| 상품·가격 업데이트 | Push (배치) | 변경 발생 시 + 1일 1회 전체 동기화 | 셀메이트CM → 스캔페이 |
| 일별 판매 집계 | Pull (배치) | 매일 새벽 2시 | 통합 대시보드용 |
| 재고 전체 대조 | Pull (배치) | 매일 새벽 3시 | 드리프트(불일치) 감지 후 알림 |
| 포인트 이벤트 | Push (실시간) | 즉시 | 결제 완료 시 동기화 |
| 쿠폰 유효성 | Pull (실시간) | 요청 시 즉시 | POS에서 쿠폰 코드 스캔 시 |

### 4.3 Webhook 수신 처리 공통 패턴

- 엔드포인트 인증: HMAC-SHA256 서명 검증 (스캔페이 Secret Key 기반)
- 멱등성 보장: `event_id` 기준 중복 이벤트 무시
- 실패 대응: 셀메이트CM 수신 실패 시 스캔페이 측 3회 재전송 (5분·15분·1시간 간격), 이후 셀메이트CM 측 폴링 보조 (1시간 주기)
- 이벤트 로그: 모든 Webhook 수신 이력 `pos_webhook_events` 테이블에 보관 (90일)

### 4.4 오프라인(네트워크 단절) 대응

- POS 로컬 큐(Local Queue): 인터넷 단절 시 POS 기기 로컬 스토리지에 판매 이벤트 임시 저장
- 재연결 시 배치 동기화: POS가 셀메이트CM에 미전송 이벤트 일괄 전송
- 충돌 처리: 동기화 중 재고가 음수가 되는 경우 → 알림 발송 + 판매자 수동 조정 요청
- 클릭앤콜렉트 운영 조건: POS 오프라인 상태에서는 해당 매장의 클릭앤콜렉트 일시 중단 처리

### 4.5 보안

- POS API 키: DB 레벨 AES-256 암호화 저장
- 스캔페이 OAuth 토큰: refresh token 자동 갱신, 만료 알림
- 매장별 권한 격리: 판매자 A의 POS 데이터가 판매자 B에게 노출되지 않도록 `store_id` + `merchant_id` 복합 인덱스 기반 쿼리

## 5. 우선순위 (P0~P2)

| 기능 | 우선순위 | 근거 |
|------|---------|------|
| 셀메이트 스캔페이 OAuth 연동 설정 화면 | **P1** | 오프라인 연계 기능 전체의 진입점. 클릭앤콜렉트 전제 조건 |
| 스캔페이 POS 판매 이벤트 Webhook 수신 | **P1** | 오프라인 재고 차감의 핵심 경로 |
| 재고 원장 차감 처리 (Webhook 기반) | **P1** | `offline-online-integration.md` P0 재고 원장과 연동 |
| 픽업 대기 주문 → POS 전송 | **P1** | 클릭앤콜렉트 운영 필수 |
| 셀메이트CM → 스캔페이 상품·재고 Push | **P1** | POS에서 판매할 상품 목록 동기화 |
| POS 어댑터 레이어 설계 (base.py) | **P1** | 타사 POS 확장을 위한 구조적 기반. 스캔페이 연동 시 함께 설계 |
| 일별 판매 집계 배치 (통합 대시보드 연계) | **P1** | 통합 정산 대시보드(`unified-payment-dashboard.md`)에 오프라인 매출 포함 |
| 도도포스 연동 | **P2** | 판매자 요구 데이터 수집 후 착수. 어댑터 패턴 확장으로 구현 |
| 포스뱅크 연동 | **P2** | 시장 조사 + 판매자 요구 확인 후 착수 |
| 재고 전체 대조 배치 (드리프트 감지) | **P2** | 안정성 고도화. MVP에서는 수동 조정 UI로 대체 가능 |
| 얼리페이 및 기타 POS 연동 | **P3** | 시장 점유율·판매자 요구 확인 후 결정 |
